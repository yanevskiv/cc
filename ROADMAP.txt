  / __/ __|
 | (_| (__   Roadmap: own the whole toolchain, C -> ELF
  \___\___|

Goal:

    Own the entire toolchain ourselves -- cc, as, ld, ar, objcopy, objdump
    -- with no gcc, no GNU binutils and no libc.  A C source becomes a
    static, freestanding ELF executable produced start to finish by tools
    we wrote.

    Unlike the first sketch (cc emits a static executable directly), we now
    go through real, inspectable ELF objects, the way a normal toolchain
    does:

        foo.c  --cc-->  foo.o  (ET_REL) --\
                                           ld --> a.out  (ET_EXEC)
        libc.c --cc-->  libc.o (ET_REL) --/

    Objects are honest ET_REL files: a section header table, .symtab,
    .strtab and .rela.* relocations with real R_X86_64_* types and proper
    symbol binding / visibility.  Our own ld resolves symbols across them
    and applies the relocations.  Because we emit standard ELF, each tool
    is checkable against its GNU counterpart: our .o should link with the
    system ld, and our ld should link a system-produced .o.

    Out of scope (for now): linker scripts, dynamic linking / shared
    objects, PIE.  Only static linking, a -r (relocatable) output mode, and
    simple section placement (-place=SEC@ADDR).

Naming conventions:

    Ast_*   abstract syntax tree (front end)
    Gen_*   code generation (AST -> instruction list)
    Asm_*   instruction list (IR): operand / instruction model and builder
    Enc_*   machine-code encoder: Asm_* -> Elf_* object bytes (ET_REL)
    Txt_*   textual assembly codec: Asm_* <-> .s source, split by syntax
            (AT&T now, room for Intel)
    Elf_*   generic ELF object library: byte buffers, sections, symbols,
            relocations, and read/write of ET_REL and ET_EXEC -- no policy,
            no exit(), and no section is named by the library (Stage 5b)
    Link_*  linker policy over Elf_*: merge sections, resolve symbols
    Rel_*   machine relocation application (x86-64 R_X86_64_* semantics)
    As_*    assembler driver: as.c's int main() and static helpers (the
            file has no header), argument parsing over Txt_*/Enc_*
    Ld_*    linker driver: ld.c's int main() and static helpers (the file
            has no header), argument parsing over Link_*
    Str_*   string helpers (incl. <regex.h> wrappers for the assembler)

Tools (binaries):

    build/bin/cc   .c -> .s (-S) | .o (-c) | a.out (default: compile + link)
    build/bin/as   .s -> .o (ET_REL)
    build/bin/ld   .o ... -> a.out | .o (-r)
    [horizon]      ar (.a archives), objcopy, objdump

    cc.c and ld.c each own an int main(); the common modules (elf.o,
    asm.o, str.o, ...) link into both.  The Makefile builds shared objects
    once and a per-tool main on top.

Module dependency:

    Gen_*  ->  Asm_*  ->  { Txt_Write  -> .s              (debug / -S)
                            Enc_Object -> Elf_* (ET_REL)  (-c)  }
    As_*   ->  Txt_Parse (.s -> Asm_*) -> Enc_Object -> Elf_* (ET_REL)
    Ld_*   ->  Link_*/Rel_*  ->  Elf_* (read ET_REL, write ET_EXEC/ET_REL)

--------------------------------------------------------------------------

Stage 1 -- Asm_* instruction IR (no behavior change)   [DONE]

    Replace Gen_EmitLine() (printf straight to file) with structured
    builder calls that append to an in-memory instruction list.  A text
    printer walks the list to reproduce the .s, so the as/ld pipeline is
    untouched.  Pure internal refactor.

    New files: h/asm.h, src/asm.c

Stage 2 -- Asm_* encoder + direct static ELF (trivial program)   [DONE]

    A second consumer of the same list: Asm_Encode() walks the items and
    emits machine-code bytes into an Elf_* image; Elf_Finish() serializes
    a standalone static ET_EXEC.  Branches write a rel32 placeholder + a
    fixup, resolved internally once vaddrs are known.  No section header
    or symbol table -- just ELF header + one PT_LOAD + bytes.

    New files: h/elf.h, src/elf.c

    Done (met): `cc ret42.c -o a.out && ./a.out; echo $?` prints 42, no
    as/ld/gcc.  This direct path is the bootstrap; it stays as the default
    until ld (Stage 5) takes over, then becomes a debug curiosity.

Stage 3 -- ELF relocatable objects (cc -c -> .o)   [DONE]

    Grow Elf_* from "write one finished executable" to "write a real
    relocatable object", and add `cc -c`.  The encoder stops resolving
    branch/data fixups itself and instead records relocations against
    named symbols for the linker to resolve later.

    Elf_* gains, on the write side:
        - section header table (.text, .rodata, .symtab, .strtab,
          .rela.text, .rela.rodata, .shstrtab)
        - a symbol table: STT_FUNC / STT_OBJECT / STT_NOTYPE, binding
          STB_LOCAL vs STB_GLOBAL (file-locals like .L labels stay LOCAL),
          visibility STV_DEFAULT, and SHN_UNDEF for references with no
          local definition (e.g. a call to putchar)
        - relocations: R_X86_64_PC32 / PLT32 for call/jmp/rip-relative,
          R_X86_64_32S / 64 for absolute data addresses, with addends

    cc grows -c (emit foo.o, ET_REL).  Default output and -S are unchanged.

    Done when: `cc -c foo.c -o foo.o` yields a valid ET_REL --
    `readelf -ar foo.o` and `objdump -dr foo.o` agree with it, and the
    *system* ld links our foo.o into a working executable (correctness
    check against a known-good linker).

Stage 4 -- Elf_* link core (link two or more objects)   [DONE]

    Teach Elf_* the other half: read ET_REL objects back in and combine
    them.  Merge same-type sections, assign addresses, build one global
    symbol table, resolve every SHN_UNDEF against a definition in another
    object, apply each relocation, and emit a static ET_EXEC.

    Elf_* gains, on the read/link side:
        - ELF object reader (parse Ehdr / section headers / symtab /
          strtab / rela)
        - section merge with per-input base offsets; rebase symbol values
          and relocation sites into the merged image
        - global symbol resolution (multiple-definition / undefined-symbol
          diagnostics)
        - relocation application for the R_X86_64_* types above

    Exercised this stage via a small internal harness / temporary hook;
    the user-facing driver is Stage 5.

    Done when: in the harness, Elf_* links foo.o + bar.o into an a.out
    that runs and returns the expected status, no system ld involved.

    Done (met): Elf_Link() is the read/merge/resolve/relocate core, now
    driven by the Stage 5a ld (see `make stage5`).  Cross-check: the system
    ld also links our two objects into a working 42.

Stage 5a -- ld driver (build/bin/ld)   [DONE]

    Wrap the Stage 4 link core in a real tool.  src/ld.c provides ld's own
    int main() and argument parsing; all ELF work stays in Elf_*.  The
    Makefile splits into shared objects + per-tool mains and starts
    building build/bin/ld alongside build/bin/cc.

    ld command line (deliberately tiny):
        ld a.o b.o -o a.out                       static link -> ET_EXEC
        ld -r a.o b.o -o ab.o                      merge, keep relocs -> ET_REL
        ld -place=text@0x08000000 -place=data@0x20000000 a.o -o a.out
                                                   explicit section placement

    Done when: `cc -c a.c && cc -c b.c && ld a.o b.o -o a.out && ./a.out`
    works with no system ld; -r round-trips (its output relinks); -place
    is honored; and our ld also links a .o produced by the system as/cc.

    Done (met): src/ld.c provides ld's main and arg parsing (-o, -e, -r,
    -place=SEC@ADDR, object inputs) inline; all ELF work stays in Elf_Link.
    The Makefile is split into shared LIB_OBJS + per-tool mains (cc.o, ld.o)
    and builds both build/bin/x86_64-cc and build/bin/x86_64-ld.  Elf_Link
    branches: -r merges the symbol tables (globals unified by name, locals
    kept per object) and rebases relocations into one ET_REL; otherwise it
    resolves and writes an ET_EXEC, with -place giving each output section its
    own PT_LOAD at the requested address.  The ET_REL writer is shared with
    cc -c (Elf_EmitSymtab + Elf_WriteRelFile).  `make stage5` checks the
    static link, the -r round-trip and -place; cross-checks: GNU ld links our
    exec and -r objects, and our ld links a GNU-produced .o -- all return 42.
    (cc --start-obj remains as the crt stand-in for _start until Stage 7.)

    Caveat carried into 5b: it works, but elf.c is three half-libraries
    fused at file-scope globals with ".text/.rodata only" baked in -- so
    -place only knows text/data and the reader/writer/encoder cannot share
    a symbol or a section.  See ELF.txt (autopsy) and ELF_REWRITE.txt.

Stage 5b -- ELF rewrite: generic ELF library + link/reloc split   [DONE]

    Replace the Stage-5a ELF code with one in-memory ELF model and small,
    single-purpose function families, then lift all linker policy out of
    the container.  No new feature -- a structural rewrite so adding a
    section, placing one by name, or sharing code between reader and writer
    stops fighting the design.  Full spec in ELF_REWRITE.txt.

    elf.c becomes a generic, section-agnostic, exit()-free library over one
    model: an Elf is a list of Elf_Sec (each owning its byte buffer and its
    own relocation list) plus one Elf_Sym table.  Read and write are
    inverses over that model, split into families by the noun they own:
        Elf_Buf*      growable byte buffers (no ELF knowledge)
        Elf_Section*  the section list: add / find-by-name / place / iterate
        Elf_Symbol*   the symbol table: add / find / iterate
        Elf_Rela*     relocations, attached to the section they patch
        Elf_Read*     bytes -> Elf        Elf_Write*  Elf -> bytes
    In memory there are names and pointers, never indices: .strtab,
    .shstrtab, st_shndx, sh_link/sh_info and the r_info symbol index exist
    only inside Elf_Write (synthesized) and Elf_Read (dissolved).  The
    library names no section -- which sections exist is the caller's data,
    and placement is just sec_addr on a named section, so -place=NAME@ADDR
    works for any section.

    Linker policy moves into its own modules, as clients of the library:
        Link_*   (src/util/link.c)        merge same-named sections, resolve
                                          symbols (unify globals, keep
                                          locals), diagnose multiple /
                                          undefined definitions
        Reloc_*  (src/arch/x86_64/reloc.c) apply relocations -- the only
                                          code that interprets R_X86_64_*
                                          (S+A-P, S+A, ...)
    cc's back end builds an Elf and calls Elf_Write; ld parses arguments,
    calls Link_Run, and calls Elf_Write.  Neither owns ELF internals.

    Staged so the tree keeps building (shims first, deletions last):
      1. new Elf_Buf/New/Section/Symbol/Rela + Elf_Write(ET_REL); old
         Elf_FinishRel becomes a shim over it -- `make test` unchanged.
      2. Elf_Read + Link_*/Reloc_* on the new model; Elf_Link a shim.
      3. ET_EXEC + PT_LOAD synthesis in Elf_Write; route the Stage-2 direct
         output and the linker's exec output through it; drop the shims and
         the file-scope globals.
      4. port cc/ld to the new API; remove the last .text/.rodata
         assumptions.

    Done when: cc -c, ld, ld -r and -place still pass `make stage5` and the
    GNU cross-checks (objects link, exec runs, all return 42) with no
    behavior change; elf.c names no section and calls no exit(); link/reloc
    policy lives in link.c/reloc.c; and -place accepts an arbitrary section
    name -- proven by placing a third, differently-named section.

    Done (met): elf.c is the generic library (Elf_Buf/Section/Symbol/Rela/
    Read/Write over an opaque Elf of Elf_Sec + Elf_Sym); it names no section
    and never calls exit().  Link policy is in src/util/link.c (Link_Run /
    Link_Merge / Link_Exec) and relocation in src/arch/x86_64/reloc.c
    (Reloc_Apply, the only reader of R_X86_64_*).  cc's back end builds the
    object in asm.c (Asm_x86_64_BuildObject) and writes via Elf_Write; the
    freestanding exec is just Link_Exec over that one object.  ld parses args
    and calls Link_Run + Elf_Write.  `make stage5` passes (static link, -r
    round-trip, -place) and the GNU cross-checks hold: readelf reads our
    objects, system ld links them to a 42, and -place works by literal ELF
    section name (.text/.rodata), not only the text/data aliases.

Stage 6 -- as (assembler: .s -> .o)

    The encoder and the ET_REL writer already exist (Stages 2-3), so the
    assembler is mostly a front end: parse AT&T text into the same Asm_*
    instruction list, then reuse Enc_* + the Elf_* object writer.  It
    is the inverse of Txt_*'s writer, and because cc -S emits exactly the
    syntax as parses, the two round-trip by construction.

    Parsing is line-oriented and regex-driven: extend Str_* with thin
    <regex.h> wrappers (Str_RegexMatch, Str_RegexExtract), then match the
    three line shapes -- `label:`, `.directive args`, `mnemonic op, op`.
    Operands cover the forms the printer emits: %reg, $imm, disp(%base),
    sym(%rip), sym.  A new Txt_x86_64_Att_Parse (in txt.c, paired with
    Txt_x86_64_Att_Write) turns text into the IR; src/as.c (As_*) is just
    main + file I/O + argument parsing.

    Directive set is small -- what cc -S emits, plus a little:
        .text .data .section .rodata .globl/.global .extern
        .string .byte .word .long .quad .skip .align .file
    No .macro, no .if, no AVX-512 -- only the closed instruction set we
    already encode.  Undefined symbols (e.g. call printf) become
    R_X86_64_PLT32/PC32 relocations, exactly as cc -c records them.

    Done when: `cc -S foo.c -o foo.s && as foo.s -o foo.o` yields an object
    equal in effect to `cc -c foo.c -o foo.o` (both link and run the same),
    and our as assembles a hand-written runtime .s; cross-check: the system
    as accepts the same .s, and our as feeds the system ld.

Stage 7 -- freestanding libc as a linked object

    With objects, a linker and an assembler in hand, the runtime is just
    more input.  Write the crt in assembly -- _start and the one
    irreducible byte-out primitive putchar -- and assemble it to crt0.o
    with our own as.  The rest of libc is written in our own C and compiled
    to libc.o.  The standard library grows with the language: putd(int)
    now; printf, puts, char* helpers as char / pointers / varargs land.

    Done when: `cc putd.c -o putd && ./putd` (cc drives cc -c + as + ld
    over the crt/libc objects) prints a number with nothing but our tools.

Stage 8 -- cc as the driver; retire system as/ld

    Default cc output becomes a fully linked ET_EXEC: cc compiles each
    input to a (temp) .o and invokes our ld.  -S dumps text asm, -c stops
    at the object.  build.mk collapses to a single rule:
        user/%: user/%.c ; $(CC) -o $@ $<
    Remove the system as/ld from the pipeline.

    Done when: nothing outside our own tools touches the .c -> executable
    path.

--------------------------------------------------------------------------

Horizon (after the core pipeline is self-hosting):

    ar        static archives, so libc.o becomes libc.a and ld pulls only
              the objects a program references.
    objcopy   section/format surgery on our own objects.
    objdump   disassemble + dump our own objects (closes the loop: every
              tool checkable without GNU binutils).

Notes:

    - Stages stay independently shippable: Stage 2's direct-ELF path keeps
      working until Stage 5's ld replaces it as the default.
    - We emit standard ELF on purpose -- cross-checking each tool against
      its GNU counterpart (our .o + system ld, our ld + system .o) is the
      cheapest correctness test we have until objdump exists.
    - Single static link only: no linker scripts, no dynamic linking, no
      PIE.  -place covers the one layout knob we actually want.
    - <elf.h> provides Elf64_Ehdr/Shdr/Sym/Rela; hand-defined to stay
      dependency-free.
