 | (_| (__   Style guide: how to write code and text in this project
  \___\___|

The point of these rules is that every file reads the same way.  When in
doubt, copy the shape of the surrounding code rather than introducing
a new convention.

--------------------------------------------------------------------------

C code:

    Modules:

        Code is grouped into modules, each with a PascalCase prefix
        naming its area:

            Ast_*   abstract syntax tree (front end)
            Gen_*   code generation (AST -> instruction list)
            Asm_*   instruction IR: operand/instruction model and builder
            Enc_*   machine-code encoder (instruction list -> ELF object)
            Txt_*   textual assembly codec (instruction list <-> .s source)
            Elf_*   ELF image: section buffers, symbols, object output
            Rel_*   architecture relocation types and patching
            Link_*  static linker policy: merge, resolve, place
            Str_*   string helpers

        An architecture-specific module carries an arch infix:
        Asm_x86_64_*, Enc_x86_64_*, Txt_x86_64_*, Gen_x86_64_*,
        Rel_x86_64_*.

        Every function and global is Prefix_PascalCase, e.g. Gen_EmitExpr,
        Asm_EmitAdd, Ast_Program.  A module's names live in h/<mod>.h, and
        the header is meant to map the source: it declares every function
        the .c file defines, and the principal structs, in the same order,
        so a reader sees the whole shape of the .c from the .h alone and is
        never surprised by a function the header did not mention.  Every
        function is declared -- none is hidden behind `static`.  A module's
        file-scope state (its static variables) and private #defines stay
        in the .c; the header maps the interface, not the storage.

        The tool drivers are the exception.  cc.c, as.c and ld.c are each
        an int main() plus a few private helpers over the modules above,
        and have no header of their own; those helpers (As_* in as.c, Ld_*
        in ld.c) are static, since a driver is short enough that the file
        itself is the overview a header would otherwise give.

        Struct fields take a short lower-case tag derived from the type,
        used on every field of that struct:

            Ast_Node  -> an_kind, an_lhs, ...      Asm_Item    -> ai_op,
            ...  Ast_Var   -> av_name, av_offset, ...   Asm_Operand ->
            ao_kind, ...  Ast_Func  -> af_name, af_body, ...

    Comments:

        In a `.c` file, every function and every global variable
        (including macros) has one short `//` line directly above it, no
        blank line in between, saying what it is -- just like cc.c.  A
        `.h` file lists those same declarations grouped for overview:
        gather related ones, separate the groups with a blank line and put
        one `//` line above each group, the way str.h and elf.h do.  The
        header names every declaration the file exports but carries only
        the group label, never a copy of each function's description --
        that descriptive comment lives once, in the .c.

        A header group comment only names the group; it does not describe
        it, and it carries no trailing period because it is a label, not a
        sentence.  Write "Code emission helpers" not "Code emission
        helpers: expressions, statements and the stack." -- the prototypes
        below it already say what each one does.

        Every `//` comment is at most one line, in headers and sources
        alike, and that includes the comment above a struct or typedef.
        If a sentence does not fit, shorten the thought rather than
        wrapping it across two lines.  Enum and struct members may have a
        trailing `//` note.  Do not write banner or section comments such
        as `// --- text back end ---`; the per-item and per-group
        comments carry the file.

        Inside a function body, a comment may only divide the code
        into named phases (e.g. "Front end" / "Back end" in main, or
        "Prologue" / "Epilogue").  Never explain how or why the code works
        -- no notes on register state, ABI rules or what a line does;
        the code itself must be enough.  Replace magic numbers with a
        named #define rather than a comment that explains the constant.

    Layout:

        K&R braces: the opening brace of a function sits on its own
        line; for every other block it stays on the statement line.
        Four-space indent, no tabs.  Functions are always written
        out in full -- never collapse a body onto one line, even a
        single-statement helper.  One blank line between functions.

        Names: avoid single-letter variable and parameter names.
        Give a variable the name of what it holds -- buf for an
        Elf_Buf, elf for an Elf, sec for an Elf_Sec, rel for an
        Elf_Rela, sym for an Elf_Sym.  The only exceptions are short
        loop counters and counts: i, n and the like are fine.

        One function per thing.  Prefer many small, named functions
        over one that switches on a tag.  Each x86 instruction has
        its own builder (Asm_EmitAdd, Asm_EmitMov...) that fills
        an item directly, rather than a single generic Asm_Emit(op,
        ...) dispatcher.  The only place a big per-opcode switch is
        allowed is the machine-code encoder.

    Formatted strings:

        A function that stores a name or label takes a printf-style
        format and its own varargs (Asm_EmitJmp(".L.end.%d", n)),
        formats it with Str_VFormat, and owns the result -- freeing
        it later.  Callers pass literals and plain values; they never
        build the string or juggle ownership.  Mark such prototypes
        __attribute__((format(printf ...))) so the compiler checks
        the arguments.

Text files (README, ROADMAP, this file):

    Plain ASCII, hard-wrapped to about 74 columns.  Open with the
    four-line ___ ___ banner and a one-line title.  Sections are a short
    heading followed by body text indented four spaces.  Use a row of
    dashes to separate major parts.  No Markdown.
