==============================================================================
 cc - a tiny C compiler:  HOW IT WORKS
==============================================================================

cc is a small C compiler built with flex and bison.  It accepts a subset of
C, lowers it to x86-64 assembly, and hands that assembly to the system
assembler/linker to produce an ELF executable.

The aim is "tcc level or less": cc itself does the real work (scanning,
parsing, code generation); it is not in the gcc/clang weight class and is not
trying to be.  It is meant to be small enough to read in one sitting and to
serve as a base you can grow your own language on top of.


== 1. The pipeline ==

  source.c
     |   flex     (lex/lexer.flex)        characters  -> tokens
     v
  token stream
     |   bison    (lex/parser.y)          tokens      -> AST
     v
  AST              (h/ast.h, src/ast.c)
     |   codegen  (src/codegen.c)         AST         -> x86-64 assembly
     v
  out.s
     |   gcc / $CC                        assemble + link with libc
     v
  a.out            (a normal ELF executable)

Only the last stage uses an external tool, and only as an assembler and
linker.  All of the compiler-specific logic - what the language means and how
it turns into machine instructions - lives in this repository.


== 2. The stages in detail ==

2.1 Scanner  --  lex/lexer.flex
    flex turns this file into out/lex.yy.c.  It recognises keywords (int, if,
    for, return, ...), identifiers, integer and character constants, string
    literals, and operators.  Whitespace and // and /* */ comments are
    skipped.  Each token may carry a value (a number or a string) which is
    passed to the parser through the `yylval` union.  String and char escape
    sequences (\n, \t, \\, ...) are decoded here.

2.2 Parser  --  lex/parser.y
    bison turns this grammar into out/parser.tab.c plus out/parser.tab.h (the
    shared token definitions).  The grammar is deliberately *flat*: instead of
    a tower of non-terminals, operator precedence and associativity are stated
    once in the %left / %right declarations near the top, and bison resolves
    the ambiguity.  Each grammar rule has an action in { } that builds a piece
    of the AST.  The parser also tracks the variables of the function it is
    currently reading so that uses of a name can be linked to its declaration.

2.3 AST  --  h/ast.h, src/ast.c
    The tree is made of `Node` structs.  A node has a `kind` (ND_ADD, ND_IF,
    ND_CALL, ...) and a handful of generic fields that different kinds reuse:

        lhs, rhs          operands of an operator or an assignment
        cond, then, els   an if statement
        init, cond,
          inc, body       a for / while loop
        funcname, args    a function call
        val / str_idx /
          var             a literal or a variable reference
        next              the next node in a list (statements, arguments)

    A `Function` records its name, parameters, body, and the list of every
    local variable.  A `Var` is one local; its stack `offset` is filled in
    later by the code generator.  Whole programs are a linked list of
    Functions, rooted at the global `program`.

2.4 Code generator  --  src/codegen.c
    Walks the AST and prints x86-64 assembly in AT&T syntax following the
    System V AMD64 ABI (the standard Linux calling convention).  See section 4.

2.5 Driver  --  src/main.c
    Parses the command line, runs the front end (yyparse), writes the assembly
    to a temporary .s file, then invokes the system C toolchain to assemble and
    link it.  The toolchain defaults to `gcc` and can be overridden with the
    $CC environment variable.


== 3. The directory layout ==

    lex/    flex (.flex) and bison (.y) grammar sources
    src/    compiler C sources          h/   compiler C headers
    test/   sample program (test.c)
    mk/     build.mk, which becomes build/Makefile
    txt/    documentation (this file)
    out/    intermediates: lex.yy.c, parser.tab.c/.h, *.o
    build/  the deliverables only: cc, test.c, Makefile

Running `make` at the project root produces three files in build/:

    cc          the compiler
    test.c      a copy of the sample program
    Makefile    a copy of mk/build.mk

so that `make -C build` compiles test.c with ./cc.  `make clean` removes both
out/ and build/.


== 4. How code generation works ==

cc uses the simplest scheme that works: a *stack machine*.

  - Every expression computes its result into the register %rax.
  - When an operator needs two values, the generator emits code for the right
    operand, `push`es %rax onto the hardware stack, emits the left operand
    (again into %rax), then `pop`s the saved value into %rdi.  Now %rax holds
    the left operand and %rdi the right, and the operator is a single
    instruction or two (add, sub, imul, idiv, cmp + setcc, ...).
  - Local variables live on the stack at negative offsets from %rbp.  Reading
    a variable computes its address with `lea` and then loads it; assigning
    computes the address, evaluates the right-hand side, and stores.
  - Function calls follow the ABI: the first six integer arguments go in
    %rdi, %rsi, %rdx, %rcx, %r8, %r9.  Before a `call` the stack is realigned
    to a 16-byte boundary when needed, and %al is set to 0 (the count of
    vector registers, required by variadic functions like printf).
  - Each function gets a prologue (save %rbp, reserve stack space, spill
    incoming parameters to their slots) and an epilogue (restore and `ret`).
    A function that runs off the end returns 0.
  - String literals are emitted in the .rodata section as raw .byte data, so
    any contents survive intact.

The generated assembly is intentionally naive - lots of redundant pushes and
pops - because the goal is clarity, not speed.  An optimiser is one of the
natural next projects.


== 5. What the language currently supports ==

  types        int, char, void, const and pointer syntax are parsed; every
               value is handled as a 64-bit integer (pointers included)
  functions    definitions and prototypes; up to 6 parameters; recursion
  statements   { blocks }, if / else, for, while, return, expression and
               empty statements, local declarations (with optional initialiser)
  operators    + - * / %, == != < <= > >=, && ||, unary - and !, assignment =
  calls        ordinary and variadic (printf works)
  literals     decimal and 0x hex integers, char constants, string literals


== 6. What it does NOT support (yet) ==

  - global variables, arrays, structs, unions, enums, typedef
  - a real type system (sizes, signedness, implicit conversions)
  - break and continue (the keywords are even tokenised, but there is no
    grammar rule for them yet - a good first extension)
  - the preprocessor (#include, #define)
  - any optimisation

See EXTEN.txt for a step-by-step guide to adding your own syntax.
