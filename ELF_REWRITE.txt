 | (_| (__   ELF, rewritten: a generic ELF object library
  \___\___|

Why rewrite at all:

    The current elf.c (see ELF.txt) is not one library, it is three
    half-libraries fused at the globals: an encoder that writes through
    file-scope state, a reader that hangs everything off Elf_Obj, and a
    linker wedged between them.  Each invented its own notion of "symbol"
    and "section", and the whole thing hard-codes that an object contains
    exactly .text and .rodata.  Adding a section, placing a section by
    name, or reusing the reader from the writer all fight the design.

    This document specifies the replacement: one in-memory ELF model and
    a set of small function families that each own a single concept.  No
    consumer (codegen, assembler, linker) keeps ELF state of its own; they
    all build, inspect and emit the same Elf object.

--------------------------------------------------------------------------

DESIGN PRINCIPLES

    1. One model, used both ways.  Reading a file and writing a file are
       inverses over the *same* struct.  Elf_Read parses bytes into an
       Elf; Elf_Write serializes an Elf back to bytes.  There is no
       separate "encoder state" and "loaded object".

    2. The library is generic.  It never names a section.  It knows about
       headers, sections, symbols, relocations and string tables as ELF
       concepts; it does not know that programs tend to have ".text".
       Which sections exist, what they are called and where they go is
       entirely the caller's data.

    3. Names, not offsets, in memory.  In the model a symbol's section is
       a pointer and a name is a char*.  String tables (.strtab,
       .shstrtab) and the numeric indices that knit ELF together
       (st_shndx, sh_link, sh_info, the symbol index in r_info) do not
       exist in memory: they are *synthesized* by Elf_Write and *dissolved*
       by Elf_Read.  Callers never compute an index or a string offset.

    4. One concept per function family.  Prefix = the noun it owns:
           Elf_Buf*       growable byte buffers (no ELF knowledge)
           Elf_Section*   the section list: add / find / place / iterate
           Elf_Symbol*    the symbol table: add / find / iterate
           Elf_Rela*      relocations, attached to the section they patch
           Elf_Read*      bytes  -> Elf
           Elf_Write*     Elf    -> bytes
       Plus Elf_New / Elf_Free for the object itself.

    5. The library has no policy.  It does not resolve symbols, apply
       relocations, decide load addresses, or call exit().  Those are the
       linker's job (a separate module, Link_*/Reloc_*), built *on top* of
       this library.  elf.c stays a dumb, total, reusable container.

--------------------------------------------------------------------------

THE DATA MODEL  (h/util/elf.h)

    A growable byte buffer -- the only storage primitive.

        typedef struct {
            uint8_t *eb_data;
            size_t   eb_len;
            size_t   eb_cap;
        } Elf_Buf;

    One symbol.  sym_sec == NULL means undefined (SHN_UNDEF); sym_value
    is an offset within sym_sec (or an absolute value for an ABS symbol).
    No index, no string offset -- those are write-time details.

        typedef struct {
            const char      *sym_name;   // owned by the Elf string pool
            struct Elf_Sec  *sym_sec;    // defining section, or NULL
            uint64_t         sym_value;  // offset within sym_sec
            uint64_t         sym_size;
            uint8_t          sym_bind;   // ELF_BIND_* (LOCAL/GLOBAL/WEAK)
            uint8_t          sym_type;   // ELF_TYPE_* (NOTYPE/OBJECT/FUNC..)
            uint8_t          sym_other;  // visibility
        } Elf_Sym;

    One relocation.  It lives in the section it patches (see Elf_Sec
    below) and points at the symbol it references.  rel_type is an opaque
    machine relocation number; the library never interprets it.

        typedef struct {
            uint64_t    rel_offset;   // within the patched section
            Elf_Sym    *rel_sym;      // referenced symbol
            uint32_t    rel_type;     // R_<machine>_* (opaque here)
            int64_t     rel_addend;
        } Elf_Rela;

    One section.  Any section -- the library imposes no set.  PROGBITS
    sections carry bytes in sec_data; a section may also own a list of
    relocations that target it (sec_relas).  sec_addr is 0 until a caller
    places it; that is the *only* notion of placement, and it is per
    named section, never per "bucket".

        typedef struct Elf_Sec {
            const char *sec_name;       // owned by the string pool
            uint32_t    sec_type;       // ELF_SHT_*
            uint64_t    sec_flags;      // ELF_SHF_*
            uint64_t    sec_addr;       // load address, 0 = unplaced
            uint64_t    sec_addralign;
            uint64_t    sec_entsize;
            Elf_Buf     sec_data;       // raw contents (PROGBITS)
            Elf_Rela   *sec_relas;      // relocations patching THIS section
            size_t      sec_nrelas;
        } Elf_Sec;

    The object.  A header's worth of fields, a list of sections, and the
    one symbol table.  String tables are not stored: they are rebuilt on
    write and discarded on read.

        typedef struct Elf {
            uint16_t    elf_type;       // ELF_ET_REL / ELF_ET_EXEC
            uint16_t    elf_machine;    // ELF_EM_X86_64, ...
            uint64_t    elf_entry;      // entry vaddr (ET_EXEC)
            Elf_Sec   **elf_secs;
            size_t      elf_nsecs;
            Elf_Sym    *elf_syms;
            size_t      elf_nsyms;
            /* private: name/string pool, allocation bookkeeping */
        } Elf;

    (The on-disk Elf64 records -- Ehdr/Phdr/Shdr/Sym/Rela -- stay private
    to elf.c.  They appear only inside Elf_Read*/Elf_Write* and never
    leak into the model or the public header.)

--------------------------------------------------------------------------

FAMILY: Elf_Buf*   -- growable byte buffers (zero ELF knowledge)

    Every append returns the offset it began at, which is what callers
    need to record a symbol value or a relocation site.

        void   Elf_BufInit(Elf_Buf *b);
        void   Elf_BufFree(Elf_Buf *b);
        void  *Elf_BufAt(Elf_Buf *b, size_t off);   // pointer, for patching
        size_t Elf_BufByte(Elf_Buf *b, uint8_t v);
        size_t Elf_BufData(Elf_Buf *b, const void *p, size_t n);
        size_t Elf_BufU16(Elf_Buf *b, uint16_t v);  // little-endian
        size_t Elf_BufU32(Elf_Buf *b, uint32_t v);
        size_t Elf_BufU64(Elf_Buf *b, uint64_t v);
        size_t Elf_BufZero(Elf_Buf *b, size_t n);   // n zero bytes
        size_t Elf_BufAlign(Elf_Buf *b, size_t a);  // pad to a multiple of a

    This is the bottom layer -- usable by the assembler/codegen directly
    for instruction bytes, and reused by Elf_Write for every table it
    builds.  It replaces Elf_BufByte/Elf_Write32/64/Bytes/Elf_BufAppend/
    Elf_BufStr, which were five spellings of the same idea.

FAMILY: Elf object lifecycle

        Elf  *Elf_New(uint16_t type, uint16_t machine);
        void  Elf_Free(Elf *e);             // frees sections, syms, pool
        void  Elf_SetEntry(Elf *e, uint64_t vaddr);

FAMILY: Elf_Section*   -- the section list

        Elf_Sec *Elf_SectionAdd (Elf *e, const char *name,
                                 uint32_t type, uint64_t flags);
        Elf_Sec *Elf_SectionFind(Elf *e, const char *name); // NULL if none
        Elf_Sec *Elf_SectionGet (Elf *e, const char *name,
                                 uint32_t type, uint64_t flags); // find-or-add
        size_t   Elf_SectionCount(const Elf *e);
        Elf_Sec *Elf_SectionAt  (const Elf *e, size_t i);
        Elf_Buf *Elf_SectionData(Elf_Sec *s);     // append bytes here
        void     Elf_SectionAddr(Elf_Sec *s, uint64_t addr);  // placement

    Elf_SectionFind is the name lookup that the old code lacked; it is
    what makes "-place=NAME@ADDR" work for *any* section:

        Elf_SectionAddr(Elf_SectionFind(e, name), addr);

    There is no current-section cursor in the library.  A consumer that
    has a notion of "current section" (the assembler, walking .text/.data
    directives) holds its own Elf_Sec* and appends to Elf_SectionData of
    it.  That state is the consumer's, not the library's.

FAMILY: Elf_Symbol*   -- the symbol table

        Elf_Sym *Elf_SymbolAdd (Elf *e, const char *name, Elf_Sec *sec,
                                uint64_t value, uint8_t bind, uint8_t type);
        Elf_Sym *Elf_SymbolFind(Elf *e, const char *name);  // NULL if none
        size_t   Elf_SymbolCount(const Elf *e);
        Elf_Sym *Elf_SymbolAt   (const Elf *e, size_t i);

    sec == NULL records an undefined reference.  Callers never order
    symbols, never assign indices, never touch .strtab: Elf_Write sorts
    locals-before-globals, builds the string table and computes sh_info.

FAMILY: Elf_Rela*   -- relocations, owned by the section they patch

        Elf_Rela *Elf_RelaAdd (Elf_Sec *target, uint64_t offset,
                               Elf_Sym *sym, uint32_t type, int64_t addend);
        size_t    Elf_RelaCount(const Elf_Sec *target);
        Elf_Rela *Elf_RelaAt   (const Elf_Sec *target, size_t i);

    A relocation references its symbol by pointer.  On write, the symbol's
    assigned index and the target section's index produce r_info and the
    .rela.<name> section automatically; on read, sh_info and the symbol
    index are resolved back to pointers.  rel_type stays an opaque number
    end to end -- only the linker's relocation applier knows what 2 or 4
    means.

FAMILY: Elf_Read*   -- bytes -> Elf

        Elf *Elf_Read   (const char *path);          // NULL on error
        Elf *Elf_ReadMem(const void *buf, size_t n);

    Parses the header, every section header, the symbol table (resolving
    each st_name through .strtab to an owned string and each st_shndx to
    an Elf_Sec*), and every .rela.* (resolving the symbol index to an
    Elf_Sym* and sh_info to the patched Elf_Sec*).  After Elf_Read the raw
    string tables are gone; only the model remains.  Reading is the exact
    inverse of writing, so an Elf_Read . Elf_Write round-trips.

    Internals (all static): Elf_ReadEhdr, Elf_ReadShdrs, Elf_ReadStrtab,
    Elf_ReadSymtab, Elf_ReadRelas.

FAMILY: Elf_Write*   -- Elf -> bytes

        int Elf_Write    (const Elf *e, const char *path);
        int Elf_WriteFile(const Elf *e, FILE *out);

    Elf_Write owns every encoding detail the rest of the program is now
    free of:
      - assign each symbol a final index, locals before globals; build
        .symtab + .strtab; set .symtab's sh_link/sh_info;
      - for each section that has relocations, synthesize a .rela.<name>
        section with sh_link -> .symtab, sh_info -> the patched section;
      - build .shstrtab and the section header table;
      - assign file offsets honoring sec_addralign;
      - if elf_type == ET_EXEC, synthesize PT_LOAD program headers from
        the allocatable sections that have a sec_addr, with page-congruent
        file offsets (the old Elf_PlaceOffset rule), and set e_entry.

    Because placement is just sec_addr per section, ET_EXEC layout is
    fully general: one section or twenty, default or -place, same code.

    Internals (all static): Elf_WriteStrtab, Elf_WriteSymtab,
    Elf_WriteRelas, Elf_WriteShdrs, Elf_WritePhdrs, Elf_WriteEhdr.

ERRORS

    The library does not exit().  Constructors return NULL, Elf_Read/Write
    return a status, and Elf_Error(e) yields the last message.  exit() and
    user-facing wording ("undefined symbol 'x'") belong to the drivers,
    which know the input path and the program name.

--------------------------------------------------------------------------

WHAT THE CONSUMERS BECOME

    cc back end (emit an object) -- no globals, no Finish functions:

        Elf     *e  = Elf_New(ELF_ET_REL, ELF_EM_X86_64);
        Elf_Sec *t  = Elf_SectionAdd(e, ".text", ELF_SHT_PROGBITS,
                                     ELF_SHF_ALLOC | ELF_SHF_EXECINSTR);
        Elf_Buf *tb = Elf_SectionData(t);
        ... Elf_BufByte(tb, op) ...                 // encode instructions
        Elf_Sym *f  = Elf_SymbolAdd(e, "main", t, off,
                                    ELF_BIND_GLOBAL, ELF_TYPE_FUNC);
        Elf_Sym *p  = Elf_SymbolFind(e, "putchar");
        if (! p) p  = Elf_SymbolAdd(e, "putchar", NULL, 0,
                                    ELF_BIND_GLOBAL, ELF_TYPE_NOTYPE);
        Elf_RelaAdd(t, call_site, p, R_X86_64_PLT32, -4);
        Elf_Write(e, "foo.o");
        Elf_Free(e);

    The choice of section names lives here, in codegen, where it belongs.
    Elf_Reset/Elf_ChangeSection/Elf_AddSymbol/Elf_MarkGlobal/Elf_AddRel32/
    Elf_FinishRel/Elf_FinishExec all disappear.

    ld (a thin driver over a link layer described next):

        Elf *out = Link_Run(inputs, ninputs, &opts);   // see below
        Elf_Write(out, opts.output);

--------------------------------------------------------------------------

THE LINK LAYER  (separate module: src/util/link.c, Link_*/Reloc_*)

    Linking is policy, so it is NOT in elf.c.  It is a client of the
    library, and it is small because the library carries the weight.

        Elf *Link_Run(const char *const *paths, int n,
                      const Link_Options *opts);

    built from:

      Link_Merge(out, in)
          For each section of `in`, Elf_SectionGet the same-named section
          in `out` and append in's bytes (Elf_BufAlign + Elf_BufData),
          remembering the base offset so symbol values and relocation
          sites can be rebased.  No "text or rodata" decision -- sections
          merge with their namesakes, whatever they are called.

      Link_Symbols(out)
          Copy symbols across, unifying globals by name (Elf_SymbolFind),
          keeping locals, folding undefined references into definitions.
          Multiple definition / undefined symbol are diagnosed here, with
          a path, because this is where the policy lives.

      Reloc_Apply(out)            [x86-64-specific, its own file]
          For each section's relas, compute S (rel_sym's vaddr =
          sym_sec->sec_addr + sym_value), A (rel_addend), P (target
          sec_addr + rel_offset), and patch sec_data via Elf_BufAt by
          rel_type:  PC32/PLT32 -> S+A-P, 32/32S -> S+A, 64 -> S+A.
          The only module that interprets relocation numbers.

    For an executable, Link_Run sets each allocatable section's sec_addr
    (default sequential from ELF_BASE, or from opts placement), resolves
    the entry symbol into elf_entry, runs Reloc_Apply, and returns an
    ET_EXEC Elf.  For -r it skips Reloc_Apply, leaves the relas in place,
    and returns an ET_REL Elf.  Either way the *writing* is just
    Elf_Write -- the same path cc uses.

--------------------------------------------------------------------------

OLD  ->  NEW  (every current function has a home)

    Elf_BufByte/Write32/Write64/WriteBytes   -> Elf_Buf* family
    Elf_BufStr/Elf_BufAppend                 -> Elf_BufData + Elf_BufAlign
    Elf_Reset / file-scope globals           -> Elf_New / Elf_Free (no state)
    Elf_ChangeSection / Elf_Current          -> consumer holds an Elf_Sec*
    Elf_SectionBuf (TEXT/RODATA switch)       -> deleted; Elf_SectionData
    Elf_AddSymbol / Elf_MarkGlobal           -> Elf_SymbolAdd (bind is a param)
    Elf_AddRel32                             -> Elf_RelaAdd
    Elf_FinalSym / Elf_BuildFinalSyms /
        Elf_EmitSymtab / Elf_IsGlobal /
        Elf_FindFinal                        -> inside Elf_Write (symtab build)
    Elf_WriteRelFile                         -> Elf_Write
    Elf_FinishRel                            -> Elf_Write on an ET_REL Elf
    Elf_FinishExec / Elf_ResolveRels /
        Elf_TextOffset/RodataOffset/Vaddr/
        Elf_SymbolVaddr                      -> Elf_Write(ET_EXEC) + Reloc_Apply
    Elf_ReadFile / Elf_LoadObj / Elf_Obj     -> Elf_Read / Elf
    Elf_MergeSections                        -> Link_Merge
    Elf_OutVaddr / Elf_ResolveSym /
        Elf_LinkGlobal / Elf_EntryVaddr      -> Link_Symbols + Reloc_Apply
    Elf_PatchLE                              -> Elf_BufAt + Elf_Buf* writes
    Elf_PlaceOffset / Elf_Seg                -> inside Elf_Write (PT_LOAD synth)
    Elf_LinkEmitRel / Elf_LinkEmitExec /
        Elf_Link                             -> Link_Run (in link.c)

    Net: elf.c becomes a generic, exit()-free, section-agnostic ELF
    container (Buf/Section/Symbol/Rela/Read/Write).  link.c and reloc.c
    hold all the policy.  cc and ld stop owning ELF internals entirely.

--------------------------------------------------------------------------

MODULE LAYOUT

    h/util/elf.h    the model + Elf_Buf/Section/Symbol/Rela/Read/Write
    src/util/elf.c  the generic library (no policy, no exit)
    h/util/link.h   Link_Options, Link_Run
    src/util/link.c Link_Merge / Link_Symbols (link policy + diagnostics)
    src/arch/x86_64/reloc.c   Reloc_Apply (the only reader of rel_type)

    cc.c  -> codegen builds an Elf, calls Elf_Write
    ld.c  -> parses args, calls Link_Run, calls Elf_Write

STAGING THE REWRITE (so the tree keeps building)

    1. Land Elf_Buf*/Elf_New/Section/Symbol/Rela + Elf_Write for ET_REL.
       Reimplement Elf_FinishRel as a shim over it; `make test` unchanged.
    2. Add Elf_Read; rebuild the link core as Link_*/Reloc_* on the new
       model; keep Elf_Link as a shim calling Link_Run.
    3. Add ET_EXEC + PT_LOAD synthesis to Elf_Write; route Stage-2 direct
       output and the linker's exec output through it.  Delete
       Elf_FinishExec/Elf_FinishRel/Elf_Link shims and the globals.
    4. Port cc and ld to call the new API directly; remove the last
       TEXT/RODATA assumptions.  -place now takes any section name.
