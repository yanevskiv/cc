What this module is:

    elf.c is the whole ELF half of the toolchain.  It does two jobs that
    are really one job seen from both ends:

        WRITE   turn the encoder's bytes + symbols + fixups into a file
                  - Elf_FinishExec  -> a static ET_EXEC   (Stage 2)
                  - Elf_FinishRel   -> an ET_REL object   (Stage 3)

        READ    read ET_REL objects back in and combine them
                  - Elf_Link        -> ET_EXEC, or ET_REL with -r  (4-5)

    Everything else in the file is plumbing for those four entry points.
    The cc back end (Gen_*/Asm_*) only ever calls the small append API
    (Elf_WriteByte/32/64/Bytes, Elf_AddSymbol, Elf_MarkGlobal,
    Elf_AddRel32) and then one of the two Finish functions.  ld calls
    Elf_Link and nothing else.

    The format is hand-rolled on purpose -- no <elf.h>, no libbfd.  The
    structs Elf_Ehdr/Phdr/Shdr/Sym/Rela are the standard ELF64 layouts
    retyped locally so the toolchain stays dependency-free.

A reader's warning (and an apology):

    This file is more tangled than the job deserves, and it is worth
    being honest about where and why before you read it:

      - There are two symbol-collection paths that almost rhyme but do
        not share code: Elf_BuildFinalSyms (the writer, from the encoder's
        global arrays) and the inline loop in Elf_LinkEmitRel (the linker,
        from on-disk symbol tables).  Both produce an Elf_FinalSym[] for
        Elf_EmitSymtab; they could be one routine over an abstract symbol
        source, but are not.

      - "Sections" exist at two different levels of fidelity.  The writer
        and the whole link core collapse every input section into exactly
        two output buckets, TEXT or RODATA (see Elf_SectionBuf and the
        ELF_SECTION_* enum).  That is why -place only knows "text" and
        "data": there is no general section list to place, just those two
        buckets.  A real linker keeps a list of named sections and places
        any of them; doing that here would mean replacing the two static
        Elf_Buf globals with a section table threaded through every
        function below.  We chose not to, and the limitation in
        Ld_ParsePlace is a symptom of that choice, not the cause.

      - The writer keeps program state in file-scope globals (Elf_Text,
        Elf_Rodata, Elf_Symbols, ...), while the link core is fully
        re-entrant and passes Elf_Obj around.  Two different styles live
        in one file.

    None of this is changed here; this document explains the code as it
    stands so the next change is cheap.

PART 0 -- shared vocabulary (constants and structs)

    Lines ~6-65 are named constants for every magic number the format
    needs: ELF_BASE (0x400000, where the executable maps), ELF_PAGE
    (0x1000), the e_ident class/data bytes, ET_REL/ET_EXEC, machine 62
    (x86-64), PT_LOAD + its R/X flags, the SHT_* section types, the SHF_*
    section flags, the SHN_UNDEF sentinel, the STB_*/STT_* symbol
    binding+type nibbles, and the five R_X86_64_* relocation numbers we
    handle (64, PC32, PLT32, 32, 32S).

    Three macro families pack/unpack the bitfields the format stores
    packed:
        ELF_ST_INFO / ST_BIND / ST_TYPE   -- st_info  = bind<<4 | type
        ELF_R_INFO  / R_SYM  / R_TYPE     -- r_info   = sym<<32 | type
    ELF_REL_ADDEND (-4) is the addend every rel32 carries: the field
    sits 4 bytes before the end of the instruction, so "distance to the
    exact target" is S + A - P with A = -4.

    Elf_Fail(...) is the writer's panic: print "cc: error: ..." and
    exit(1).  It is used for both genuinely-internal invariants and for
    user-facing link errors (undefined symbol, multiple definition); a
    nicer tool would separate those, but one macro covers both today.

    The structs that follow are the on-disk ELF64 records:
        Elf_Ehdr   file header
        Elf_Phdr   program header (one loadable segment)
        Elf_Shdr   section header (ET_REL output / ET_REL input)
        Elf_Sym    one .symtab entry
        Elf_Rela   one .rela.* entry (explicit addend)
    plus three in-memory helpers used only while building:
        Elf_Buf    a growable byte vector (eb_data/eb_len/eb_cap)
        Elf_Symbol a defined symbol recorded during encoding
                   (name, which bucket, offset within it)
        Elf_Rel    a pending rel32 fixup (section, offset, target, type)

PART 1 -- the encoder-facing append API and its state

    State (file-scope globals, lines ~160-176):
        Elf_Text, Elf_Rodata    the two output section buffers
        Elf_Current             which one writes go to now
        Elf_Symbols/NumSymbols  defined symbols, in encounter order
        Elf_Rels/NumRels        pending rel32 fixups
        Elf_Globals/NumGlobals  names seen in a .globl

    Elf_SectionBuf(sec)
        The one place the TEXT/RODATA-only model is hard-coded: maps the
        section enum to &Elf_Text or &Elf_Rodata.  Everything that says
        "section" upstream bottoms out here in a two-way choice.

    Elf_BufByte(buf, byte)
        Append one byte, doubling capacity (from 256) when full.  The
        single primitive every other writer routine builds on.

    Elf_Reset()
        Free both buffers and zero all the counters so a fresh program
        can be encoded.  cc calls this between translation units.

    Elf_ChangeSection(sec)           set Elf_Current.
    Elf_WriteByte(byte)              append to the current buffer.
    Elf_Write32 / Elf_Write64(val)   append little-endian, byte by byte.
    Elf_WriteBytes(data, len)        append a raw run (e.g. a string).

    Elf_AddSymbol(name)
        Record name at "here": current section + current buffer length.
        No type/binding yet -- that is decided at finish time.

    Elf_MarkGlobal(name)
        Remember that name appeared in a .globl.  Binding is applied
        later by matching this list against the defined symbols.

    Elf_AddRel32(target, type)
        Reserve a 32-bit relocation slot at "here": remember the section,
        the offset, the target name and PC32-vs-PLT32.  The 4 placeholder
        bytes themselves were already written by the caller.

PART 2 -- Elf_FinishExec: the direct static executable (Stage 2)

    This is the bootstrap path: no sections, no symbol table, just a
    header + one PT_LOAD + the bytes, with every fixup resolved in place.

    Elf_TextOffset()      file offset where .text begins
                          = sizeof(Ehdr) + sizeof(Phdr).
    Elf_RodataOffset()    = TextOffset + len(.text); .rodata follows text.
    Elf_Vaddr(sec, off)   virtual address of a spot = ELF_BASE + its file
                          offset (file offset 0 maps to ELF_BASE, so file
                          and virtual addresses differ by a constant).
    Elf_SymbolVaddr(name) linear search of Elf_Symbols for name, returned
                          as a vaddr; Elf_Fail if undefined.

    Elf_ResolveRels()
        For every pending Elf_Rel: compute site = vaddr of the slot,
        target = vaddr of the symbol, and write the signed 32-bit
        delta = target - (site + 4) straight into the buffer.  This is
        the self-contained linker for the one-file case; Stage 3 replaces
        it with recorded relocations instead.

    Elf_FinishExec(out, entry)
        Resolve all rel32s, then lay out the file: filesz covers headers
        + text + rodata; the Ehdr names ET_EXEC, machine x86-64, entry =
        vaddr of `entry`, one program header; the Phdr is a single R+X
        PT_LOAD of the whole image at ELF_BASE.  Write Ehdr, Phdr, text,
        rodata.  Done.

PART 3 -- the ET_REL writer (Stage 3), shared by cc -c and ld -r

    Section indices are fixed so symbols can name them by number without
    a lookup: .text = 1, .rodata = 2 (ELF_SHNDX_TEXT / _RODATA).  .symtab
    and .strtab are 3 and 4; .rela.* and .shstrtab follow.

    Elf_FinalSym (struct)
        A symbol on its way into .symtab but before local/global ordering:
        name, whether defined here, which bucket + offset, bind, type, and
        the slot fs_index it will finally occupy.  This is the common
        currency between "collect symbols" and "emit symtab".

    Elf_BufStr(buf, name)
        Append name + a NUL to a string table buffer, return the offset it
        started at (what goes in st_name / sh_name).

    Elf_IsGlobal(name)        was name in a .globl?  (linear search)
    Elf_FindFinal(fs, n, nm)  index of nm in an Elf_FinalSym[], or -1.

    Elf_BuildFinalSyms(fs)   [writer's symbol collector]
        Two passes over the encoder state:
          1. every defined symbol -> a row; global ones get STT_FUNC if in
             text else STT_OBJECT, locals stay STT_NOTYPE.
          2. every relocation target with no local definition -> an
             SHN_UNDEF global row (this is how `call printf` becomes an
             undefined reference).
        Returns the row count.

    Elf_EmitSymtab(fs, nfs, &syms, &nsym, &strtab, &first_global)
        The shared symtab serializer (used by cc -c, ld -r, and the
        diagnostics path).  ELF requires all STB_LOCAL symbols before any
        STB_GLOBAL, with sh_info pointing at the first global -- so it
        walks fs[] twice (locals pass, then globals pass), assigns each
        row its final index, and fills an Elf_Sym[] + its .strtab.  Slot 0
        is the reserved null entry.  Defined rows get a section index +
        value; undefined rows get SHN_UNDEF.

    Elf_WriteRelFile(out, text, rodata, syms, nsym, first_global, strtab,
                     relatext, ntr, relarodata, nrr)
        The shared ET_REL file serializer.  Given finished section bytes,
        a finished symbol table and the two split relocation tables, it:
          - builds .shstrtab and the Shdr table.  .text/.rodata sit at the
            fixed indices 1/2; .symtab/.strtab at 3/4; .rela.text and
            .rela.rodata are emitted only if non-empty (their indices
            shift accordingly), then .shstrtab last;
          - .symtab's sh_link -> .strtab, sh_info = first_global; each
            .rela.* has sh_link -> .symtab and sh_info -> the section it
            patches;
          - assigns each section a file offset honoring its alignment,
            then 8-aligns the section-header table after them;
          - writes Ehdr (ET_REL, e_shoff/e_shnum/e_shstrndx set), then the
            section bodies padded to their offsets, then the Shdr table.

    Elf_FinishRel(out)       [cc -c entry point]
        Glue for the single-object case: BuildFinalSyms -> EmitSymtab,
        then turn each pending Elf_Rel into an Elf_Rela (PLT32 or PC32,
        addend -4, symbol = its fs_index) sorted into relatext/relarodata
        by section, then hand it all to WriteRelFile.

--------------------------------------------------------------------------

PART 4 -- the ET_REL reader + link core (Stages 4-5)

    From here the file is re-entrant: no globals, everything hangs off an
    Elf_Obj per input.

    Elf_Obj (struct)
        One loaded object: the raw file image (owned), typed pointers into
        it (ehdr/shdr/syms/symstr), and two parallel arrays indexed by the
        object's own section number --
            lo_outsec[s]  which output bucket section s landed in (-1 none)
            lo_base[s]    its start offset within that merged bucket
        These two arrays are how an input symbol value or relocation site
        gets rebased into the merged image.

    Elf_LinkGlobal (struct)   a resolved global: name + final vaddr.
    Elf_Seg (struct)          a placed PT_LOAD: buffer + vaddr + file off.

    Elf_ReadFile(path, &size)
        Slurp a whole file into a malloc'd buffer (Elf_Fail on open/short
        read).

    Elf_LoadObj(obj, path)
        Read the file, validate it is ELF / ET_REL / x86-64, find the
        SHT_SYMTAB section (and its linked .strtab), and fill the Elf_Obj.
        Allocate lo_outsec (all -1) and lo_base (all 0): nothing is placed
        until Elf_MergeSections runs.

    Elf_BufAppend(buf, src, len, align)
        Like Elf_BufByte in bulk: pad the buffer up to `align`, then copy
        len bytes, returning the offset they began at.  This is what makes
        merging "section after section" honor each input's alignment.

    Elf_OutVaddr(outsec, text_vaddr, rodata_vaddr)
        Trivial selector: the base vaddr of the TEXT or RODATA bucket.
        Centralized so the two-bucket assumption has one name.

    Elf_ResolveSym(o, symidx, text_vaddr, rodata_vaddr, globals, nglob)
        Turn a symbol reference into a vaddr.  A symbol defined in a merged
        section (local label or global alike) is placed directly from its
        bucket base + lo_base + st_value -- no name lookup needed.  An
        SHN_UNDEF reference is matched by name against the program-wide
        global table.  Elf_Fail if neither.

    Elf_PatchLE(at, value, width)
        Write `value` little-endian into `width` bytes.  The one byte-
        poker the relocation appliers use.

    Elf_MergeSections(objs, npaths, text, rodata)
        The layout pass.  For every allocatable PROGBITS section of every
        object, append its bytes to text (if SHF_EXECINSTR) or rodata
        (otherwise) via Elf_BufAppend, and record lo_outsec/lo_base.  After
        this, the merged buffers are final in size and every input section
        knows where it lives.  (This is also where the model decides there
        are only two output sections.)

    Elf_LinkEmitRel(objs, npaths, text, rodata, out)        [ld -r]
        Merge into a new ET_REL instead of resolving.  Builds an
        Elf_FinalSym[] from the on-disk symbol tables: globals are unified
        by name across objects (a later definition fills in an earlier
        undefined row; two definitions -> Elf_Fail), locals are kept per
        object.  A per-object symmap[i][k] records where input symbol k
        went (or -1 to drop, e.g. the null entry).  Then EmitSymtab, then
        walk every input .rela.* and rebuild each entry rebased into the
        merged section (r_offset += lo_base of the patched section, symbol
        index remapped through symmap, type + addend preserved), split into
        relatext/relarodata.  Finally WriteRelFile -- the same serializer
        cc -c uses.  This is the function that most overlaps Part 3's
        BuildFinalSyms without sharing its code.

    Elf_PlaceOffset(pos, vaddr)
        Smallest file offset >= pos that is congruent to vaddr modulo the
        page size.  A PT_LOAD requires p_offset == p_vaddr (mod page); this
        is what lets -place put a section at an arbitrary address while
        keeping the file mappable.

    Elf_EntryVaddr(entry, globals, nglob)
        Look up the entry symbol (default "_start") in the resolved global
        table; Elf_Fail if absent.

    Elf_LinkEmitExec(objs, npaths, text, rodata, opts, out)  [ld default]
        The real static linker.  Steps:
          1. Decide bucket addresses.  Default: one segment, so text sits
             at ELF_BASE + headers and rodata right after it.  Placed:
             each bucket at its requested address (Part-of-the-reason -place
             only offers text/data), defaulting an unplaced one to ELF_BASE
             / the page above text.
          2. Build the global table: every defined global symbol -> name +
             vaddr (bucket base + lo_base + st_value); duplicate name ->
             Elf_Fail.
          3. Apply relocations.  For each input .rela.* patching a merged
             section, for each entry compute P (site vaddr), S (via
             Elf_ResolveSym) and A (addend), then patch the merged buffer:
                PC32/PLT32 -> S + A - P   (32-bit)
                32/32S     -> S + A       (32-bit)
                64         -> S + A       (64-bit)
             any other type -> Elf_Fail.
          4. Resolve the entry vaddr.
          5. Emit.  Default mode: one R+X PT_LOAD covering headers + text +
             rodata at ELF_BASE (the same shape as Elf_FinishExec, only fed
             from merged buffers).  Placed mode: one R+X PT_LOAD per non-
             empty bucket, each at its Elf_PlaceOffset'd file offset; write
             Ehdr, the phdrs, then each segment's bytes padded to offset.
             (Static executables need not map their own headers, so placed
             mode simply does not.)

    Elf_Link(paths, npaths, opts, out)                  [the ld entry]
        The top-level driver Ld calls.  Load every object, MergeSections,
        then branch: -r -> Elf_LinkEmitRel; otherwise default the entry to
        "_start" and call Elf_LinkEmitExec.  Free everything.  All of ld's
        ELF knowledge is reached through this one call.

How the pieces connect (two end-to-end paths):

    cc -c foo.c:
        Gen_*/Asm_* --append API--> [Elf_Text/Rodata/Symbols/Rels/Globals]
            --> Elf_FinishRel
                  --> Elf_BuildFinalSyms --> Elf_EmitSymtab
                  --> split rels --> Elf_WriteRelFile --> foo.o (ET_REL)

    ld a.o b.o -o a.out:
        Elf_Link
          --> Elf_LoadObj (xN) --> Elf_MergeSections
          --> Elf_LinkEmitExec
                --> build globals --> apply relocations (PatchLE)
                --> Elf_EntryVaddr --> write Ehdr/Phdr/segment(s)
        (with -r: ... --> Elf_LinkEmitRel --> Elf_WriteRelFile -> ab.o)

What would make it simpler (notes for a future pass, not done here):

    - Address sections by name, not by a fixed enum.  Neither the two
      Elf_Buf globals nor a hand-written ELF_SECTION_* enum is the right
      model: both bake the section set into the source.  The file already
      carries the authoritative list -- the section header string table,
      .shstrtab, located through the header's e_shstrndx; each Shdr's
      sh_name indexes into it.  So the placement primitive should be

          void Elf_SetAddr(const char *name, uint64_t addr);

      which finds the section by walking the headers and matching sh_name
      against .shstrtab, then sets that section's address.  Then -place
      takes any section name (the user's actual point), Elf_OutVaddr /
      Elf_SectionBuf and the TEXT/RODATA special-casing disappear, and the
      "two output buckets" assumption goes with them.
    - Factor one symbol-collection routine over an abstract "symbol
      source" so Elf_BuildFinalSyms and Elf_LinkEmitRel's inline loop stop
      duplicating the unify-globals / keep-locals logic.
    - Split Elf_Fail into an internal-invariant assert and a user-facing
      link diagnostic; the latter should carry the input path and not say
      "cc:".
    - Replace the fixed ELF_MAX_* arrays in the encoder with growable
      vectors (the link core already avoids fixed caps).
